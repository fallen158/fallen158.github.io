map

map函数需要两个必填参数，第一个参数是一个函数名，第二个参数是一个可迭代的对象，如列表、元组等。

map实现的功能很简单，就是将第二个参数（iterable）中的每一个元素分别传给第一个参数（func），依次执行函数得到结果，并将结果组成一个新的list对象后进行返回。返回结果永远都是一个list。

除了传入一个可迭代对象这种常见的模式外，map还支持传入多个可迭代对象。

reduce

reduce函数的功能是对可迭代对象（iterable）中的元素从左到右进行累计运算，最终得到一个数值。接收的参数，一个函数f，一个list，但是传入的函数f必须接收两个参数，另外还可以接受第三个可选参数.第三个参数initializer是初始数值，可以空置，空置为None时就从可迭代对象（iterable）的第二个元素开始，并将第一个元素作为之前的结果。

需要注意的是reduce在python3被放进functools包啦，使用前需要导入：from functools import reduce。

filter

filter有且仅有两个参数，第一个参数是一个函数名，第二个参数是一个可迭代的对象，如列表、元组等,这个函数f的作用是对每个元素进行判断，返回True和false，自动过滤不符合条件的元素。

sorted

自定义排序函数，默认是正序排列，它可以接受一个key函数来实现自定义排序,字符串排序依照ASCII。

为什么需要AJAX？

回顾下以前是如何发送请求的：

  • 用 form 可以发请求，但是会刷新页面或新开页面
  • 用 a 可以发 get 请求，但是也会刷新页面或新开页面
  • 用 img 可以发 get 请求，但是只能以图片的形式展示
  • 用 link 可以发 get 请求，但是只能以 CSS、favicon 的形式展示
  • 用 script 可以发 get 请求，但是只能以脚本的形式运行

看上面请求方式，都有很明显的缺点，那么有没有什么方式可以实现：

  1. get、post、put、delete 请求都行
  2. 想以什么形式展示就以什么形式展示

就是AJAX。

AJAX的历史

Jesse James Garrett 讲如下技术取名叫做 AJAX：异步的 JavaScript 和 XML。

AJAX的核心是XMLHttpRequest (XHR) API。XHR 是一种用于在 Web 浏览器和 Web 服务器间传输数据消息的 JavaScript API。它支持浏览器使用 HTTP POST（将数据传到服务器）或 GET 请求（从后台服务器访问数据）。

在 2000 年，XHR 最先出现于 IE 5 中。最初是由 Alex Hopmann 编写的 Microsoft ® ActiveX® 控件，创建 XHR 是为了处理 Microsoft Outlook® Web Access，旨在解决高级（当时）前端接口和 Microsoft Exchange Server 间的交互。

IE 5 率先在 JS 中引入 ActiveX 对象（API），使得 JS 可以直接发起 HTTP 请求。随后 Mozilla、 Safari、 Opera 也跟进（抄袭）了，取名 XMLHttpRequest，并被纳入 W3C 规范。

真正的变化是几年后谷歌的Gmail服务，这一项Ajax 技术的使用使得 Web 开发界沸腾起来。它的响应能力和交互性对公众而言是全新的。

什么是路由

传统的路由指的是：当用户访问一个URL时，对应的服务器会接收这个请求，然后解析url中的路径，从而执行对应的处理逻辑。这样就完成啦一次路由并发。例如：

简单距举例说明：假如我们有一台提供Web服务的地址是：http:/127.0.0.1，而该Web服务又提供了三个可供用户访问页面，其URL是：

  • http:/127.0.0.1/
  • http:/127.0.0.1/tab1
  • http:/127.0.0.1/tab2

那么其路径就分别是 /，/tab1，/tab2。

当用户使用 http:/127.0.0.1/tab1 来访问该页面时，Web 服务会接收到这个请求，然后会解析 URL 中的路径 /tab1，在 Web 服务的程序中，该路径对应着相应的处理逻辑，程序会把请求交给路径所对应的处理逻辑，这样就完成了一次「路由分发」，这个分发就是通过「路由」来完成的。

以前路由都是后台做的，通过用户请求的url导航到具体的html页面，前端路由就是通过配置js文件，把这个工作拿到前端来做。

简单的说，就是你给我一个路径，我给你返回一个页面（后端路由）

你给我一个路径，我给你返回一个状态（前端路由）

前端路由

前端的路由和后端的路由在实现技术上不一样，但是原理都是一样的。在 HTML5 的 history API 出现之前，前端的路由都是通过 hash 来实现的。

前端路由主要做的就是两点：

  • 在页面不刷新的情况下实现url的变化
  • 捕捉url的变化，根据url更改页面内容

  • 使用路径的话肯定会跳转页面，但我们可以使用事件阻止其默认事件，使其不能跳转页面，而且不会受到其他a标签锚点的影响。

跨域问题是由于javascript语言安全限制中的同源策略造成的。同源策略是由Netscape提出的一个著名的安全策略。 现在所有支持JavaScript 的浏览器都会使用这个策略。同源策略阻止从一个域上加载的脚本去获取或操作另一个域上的文档属性。也就是说，受到请求的 URL 的域必须与当前 Web 页面的域相同。这说明浏览器隔离来自不同源的内容，以防止它们之间的操作。所谓同源是指域名，协议，端口均相同。

同源的几种情况：

  • 不同域名属于跨域，如：www.a.com和www.b.com，另外www.a.com 和www.a.com.cn 也属于不同域名。
  • 主域名和子域名（二级域名、三级域名等）跨域，如：www.a.com 和 sub.a.com属于跨域，sub.a.com 和 sub1.a.com 之间也是跨域。
  • 协议属于跨域，如：http://www.a.com 和 https://www.a.com。
  • 不同端口，如： www.a.com:80和 www.a.com:81 。
  • IP和域名属于跨域，如：123.125.106.16 和www.weibo.com。

###为什么要跨越？

我们实际上做项目的时候，不可避免地会根据项目需求进行跨站访问，子域和主域之间数据共享等，受到同源策略的影响，要满足这些需求，就要用跨域技术来实现。

JSAONP是一种由前端创建一个script请求，指向响应方，同时传一个随机参数。后端根据随机参数响应，填写数据。浏览器接收到响应后就会执行数据，请求方就知道了他要的数据。

  • 参数，得传一个随机数字过去

JSONP有一个缺点：只能get请求，不能post。

面试题：

JSONP 为什么不支持 POST

因为JSONP是通过动态创建script的，动态创建script的时候只能用get，不能用POST。

JSON和JSONP的区别

JSON和JSONP虽然只有一个字母的差别，但其实他们根本不是一回事儿：JSON是一种数据交换格式，而JSONP是一种依靠开发人员的聪明才智创造出的一种非官方跨域数据交互协议。我们拿最近比较火的谍战片来打个比方，JSON是地下党们用来书写和交换情报的“暗号”，而JSONP则是把用暗号书写的情报传递给自己同志时使用的接头方式。看到没？一个是描述信息的格式，一个是信息传递双方约定的方法。

如何写出好代码

发表于 2018-06-16 | 分类于 重构代码

学习之前，可以先学习一下如何写出无法维护的代码：

变量命名

  • 容易输入的名字。比如：Fred，asdf，a,b,c, x,y,z。

  • 有创意地拼写错误。比如：SetPintleOpening， SetPintalClosing。这样可以让人很难搜索代码。

  • 抽象。比如：ProcessData, DoIt, GetData… 抽象到就跟什么都没说一样。

  • 缩写。比如：WTF，RTFSC ……

  • 随机大写字母。比如：gEtnuMbER..

  • 使用不同的语言。比如混用英语，德语，或是中文拼音。

  • 使用重音字母。比如：int ínt

  • 使用下划线。比如：_, , _。

  • 使用字符命名。比如：slash, asterix, comma…

  • 使用无关的单词。比如：god, superman, iloveu….

  • 混淆l和1。字母l和数字1有时候是看不出来的。

  • 表里不一。比如你的界面显示叫postal code，但是代码里确叫 zipcode.

  • 把注释和代码交织在一起。

  • 在注释中撒谎。你不用真的去撒谎，只需在改代码的时候不要更新注释就可以了。

bad smell（坏味道）

有些代码可以用，但是很「臭」。哪些代码是有坏味道的

  • 表里不一的代码
  • 过时的注释
  • 逻辑很简单，但是看起来很复杂的代码
  • 重复的代码
  • 相似的代码
  • 总是一起出现的代码

代码优化的基本原则

  1. 易读性优先
  2. 如果不是性能瓶颈，就不要为了性能而改写代码
  3. 复杂性守恒原则：无论你怎么写代码，复杂性都是不会消失的
  4. 推论：如果逻辑很复杂，那么代码看起来就应该是复杂的。如果逻辑很简单，代码看起来就应该是简单的。

命名规范

  1. 词性

  • 普通变量/属性用「名词」

  • bool变量/属性用「形容词」或者「be动词」或者「情态动词」或者「hasX」

  • 普通函数/方法用「动词」开头

  • 回调、钩子函数用「介词」开头，或用「动词的现在完成时态」

  • 容易混淆的地方加前缀

  • 属性访问器函数可以用名词

  1. 一致性

    • 介词一致性

      如果你使用了 before + after，那么就在代码的所有地方都坚持使用

      如果你使用了 before + 完成时，那么就坚持使用

      如果你改来改去，就「不一致」了，不一致将导致「不可预测」

    • 顺序一致性

      比如 updateContainerWidth 和 updateHeightOfContainer 的顺序就令人很别扭，同样会引发「不可预测」
